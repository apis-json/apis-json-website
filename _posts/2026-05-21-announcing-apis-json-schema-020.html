We've published a draft of APIs.json schema 0.20, and it's open for review on GitHub. The primary focus of this release is an expanded reserved word vocabulary — the set of recognized type values for the properties array in an APIs.json file. Based on analysis of how real providers are using APIs.json in the wild, 0.20 adds 24 new reserved types and fixes a handful of naming inconsistencies that had crept in.

What Changed

The properties array is the heart of an APIs.json file. It's where you link the operational artifacts that define what it means to work with your API — the spec, the docs, the authentication guide, the pricing page, the support channel. The type value on each entry is how tooling, indexers, and AI agents understand what kind of thing is being referenced.

Until 0.20, the reserved word list was fairly conservative — it captured the most common property types but left a lot of real-world provider usage in the X- custom prefix space. We analyzed how APIs.json is actually being used across providers and promoted the most common patterns to first-class reserved words.

New Reserved Types

The additions fall into a few natural groups:

Web Presence

Website, Portal, and DeveloperPortal let you explicitly reference the provider's web presence. These are distinct from Documentation — a website or portal is the top-level entry point; documentation is a specific technical resource within it. YouTube and LinkedIn round out the social web presence category.

Developer Experience

CLI, Console, and Sandbox address the tools developers use to explore and interact with an API. A CLI is the command-line interface for an API. A Console is the interactive API explorer in a developer portal. A Sandbox is the safe testing environment. These are common enough in API operations that they deserve reserved status rather than custom extensions.

Reference Material

Examples and FAQ were both appearing commonly enough to warrant promotion. Examples is distinct from ApiExamples — the latter refers to machine-readable request/response examples in a structured format, while Examples covers human-readable examples in documentation. Summary is also added for a high-level overview resource.

Community

Community, StackOverflow, Slack, and Discord recognize that developer community has moved well beyond forums. These cover the actual places where API consumers ask questions and get help in 2026.

Lifecycle

ReleaseNotes joins ChangeLog and RoadMap. Many providers use release notes and change log to mean different things — release notes are often more narrative and linked to a specific version, while a change log is a running record of all changes. Both are now reserved.

Technical Contracts

JSONStructure, JSONLD, Vocabulary, and SpectralRules extend the contract vocabulary. SpectralRules in particular recognizes that Spectral ruleset files have become first-class API artifacts — a machine-readable description of the governance rules that apply to an API's spec.

Legal & Compliance

Compliance, License, and Training fill gaps in the legal and regulatory space that have become more important as enterprise API adoption has matured.

Social & Source Control Aliases

X is added as the current name for the platform formerly known as Twitter (keeping Twitter for backward compatibility). GitHubOrganization and GitHubRepository are added as canonical long-form aliases for the existing GitHubOrg and GitHubRepo shorthand types.

Naming Fixes

Two naming issues from earlier versions are corrected in 0.20:

  • JsonStructure is corrected to JSONStructure — consistent with the naming convention used by JSONSchema and JSONLD
  • A duplicate ApiExamples entry in the schema pattern has been removed

The Full Reserved Word List

The complete list of reserved words — including all 0.20 additions — is published at apisjson.org/format/reservedwords. The JSON Schema validation pattern is in the spec file on GitHub.

How to Participate

The 0.20 schema is in draft. Review the PR on GitHub, leave comments on any types you think are missing, misnamed, or shouldn't be reserved. We're particularly interested in feedback on the AI/agent-related types — AgentSkills, LlmsText, MockMCPServer — which were added in earlier 0.20 work and represent an area where the vocabulary is still settling.

Once the PR is merged and 0.20 is marked stable, apis.io will start indexing against the new vocabulary. Providers who are already using custom X- types for any of the newly reserved words can migrate to the reserved form at any time — both will remain valid.
